Memcache 详解

【学习源】http://blog.jobbole.com/98733/

MC被称为“分布式缓存”，实际本身并不具备分布式功能。
而是需要依赖客户端，读写缓存时用相同的路由算法和服务器列表，而达到读写同一key访问同一服务端。

路由算法

算法一：余数Hash

1. 实现
HashCode 对服务器数目取余数，例如50%3 = 1。

2. 问题
如果需要扩容，则会出现命中失败问题。

3. 解决方案
（1）在网站访问量低谷，通常是深夜，技术团队加班，扩容、重启服务器
（2）通过模拟请求的方式逐渐预热缓存，使缓存服务器中的数据重新分布

算法二：一致性Hash算法

1. 实现

  • 先构造一个长度为232的整数Hash环
  • key值根据其HashCode的位置，顺时针寻找离他最近的一个Node

2. 问题

  • 新增服务器节点，仅加粗的那段缓存失效
  • 集群中缓存服务器节点越多，增加节点带来的影响越小

Memcache实现原理

  1. MemCache将内存空间分为一组slab
  2. 每个slab下又有若干个page，每个page默认是1M，如果一个slab占用100M内存的话，那么这个slab下应该有100个page
  3. 每个page里面包含一组chunk，chunk是真正存放数据的地方，同一个slab里面的chunk的大小是固定的
  4. 有相同大小chunk的slab被组织在一起，称为slab_class

value总是会被存放到与chunk大小最接近的一个slab中。
比如slab[1]的chunk大小为80字节、slab[2]的chunk大小为100字节、slab[3]的chunk大小为128字节（相邻slab内的chunk基本以1.25为比例进行增长，MemCache启动时可以用-f指定这个比例）

内存分配与回收算法

  1. MemCache的内存分配chunk里面会有内存浪费，88字节的value分配在128字节（紧接着大的用）的chunk中，就损失了30字节，但是这也避免了管理内存碎片的问题
  2. MemCache的LRU算法不是针对全局的，是针对slab的
  3. 应该可以理解为什么MemCache存放的value大小是限制的，因为一个新数据过来，slab会先以page#为单位申请一块内存，申请的内存最多就只有1M，所以value大小自然不能大于1M了

特性总结

1、MC可以保存的item数据量是没有限制的，内存足够即可
2、MemCache单进程在32位机中最大使用内存为2G，64位机则没有限制
3、Key最大为250个字节，超过无法存储
4、单个item最大数据为1MB，超过无法存储
5、不能够遍历MC多有的item，这个操作速度相对缓慢且会阻塞其他操作
6、MC不能做永久存储，及时expire传入了0，实际在30天后也会失效

#define REALTIME_MAXDELTA 60*60*24*30
static rel_time_t realtime(const time_t exptime) {
       if (exptime == 0) return 0;
       if (exptime > REALTIME_MAXDELTA) { 
              if (exptime <= process_started) 
                     return (rel_time_t)1; 
              return (rel_time_t)(exptime - process_started); 
       } else { 
              return (rel_time_t)(exptime + current_time); 
       }
}
